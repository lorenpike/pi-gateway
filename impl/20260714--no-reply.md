# Buffered-channel `NO_REPLY` — Implementation Plan

**Date:** 2026-07-14
**Status:** Implemented

## 1. Goal

Add a gateway-level silent-completion convention for user-facing, buffered
channels. When the complete assistant text, after trimming surrounding
whitespace, is exactly `NO_REPLY`, Telegram, Discord, and future buffered chat
front-ends must stop their activity indicator and deliver no assistant message.

HTTP/SSE is intentionally excluded. It is a raw application-facing stream used
by clients such as the CLI and must continue exposing the assistant's original
deltas and lifecycle events.

The immediate implementation also changes ordinary Discord turns from
edit-in-place previews to the same buffer-until-complete model Telegram uses.

## 2. Contract

### 2.1 Match rule

Suppress external chat delivery only when:

```go
strings.TrimSpace(authoritativeFinalText) == "NO_REPLY"
```

The comparison is case-sensitive and applies to the whole response. Therefore:

| Final assistant text | Suppress? |
|---|---:|
| `NO_REPLY` | yes |
| `  NO_REPLY\n` | yes |
| `no_reply` | no |
| `` `NO_REPLY` `` | no |
| `I will use NO_REPLY when appropriate.` | no |
| `NO_REPLY.` | no |
| empty text | no; preserve the channel's existing empty-response behavior |

Do not detect the marker incrementally and do not treat it as a substring. The
authoritative value from `turn.Subscription.FinalText`, not a locally assembled
best-effort delta buffer, decides suppression.

### 2.2 Delivery classes

**Buffered client-facing channels:**

- Telegram;
- Discord ordinary messages;
- Discord application-command interactions;
- future end-user chat, SMS, or similar adapters unless explicitly documented
  as raw transports.

These channels wait until turn completion, inspect the authoritative final
text, and then either suppress or render it.

**Raw transports:**

- HTTP `/v1/prompt` SSE;
- the `wall-e msg` CLI, because it consumes HTTP/SSE;
- future event/API transports explicitly registered as raw.

Raw transports emit the original `NO_REPLY` text exactly as generated. They
still emit normal `agent_start`, `agent_end`, and `done` lifecycle events.

**Unaffected paths:**

- `wall-e send` and `/v1/send`, which bypass the agent turn;
- tool-driven media/text sends performed during a turn;
- pi's on-disk transcript and session UI source data. The marker remains part
  of the assistant transcript; only buffered external delivery is suppressed.

### 2.3 Multiple subscribers

One turn can have both raw and buffered subscribers. For example,
`wall-e msg telegram:<chat-id>` starts a Telegram delivery and also streams the
same turn to the CLI. If pi returns `NO_REPLY`:

- the CLI sees `NO_REPLY` in its raw HTTP stream;
- Telegram sends no final message;
- both subscriptions complete normally.

Suppression must therefore be a buffered-delivery policy, not a mutation of
`turn.Manager` events or `FinalText`.

## 3. Design

### 3.1 Shared buffered response helper

Add a small channel-independent helper in `src/chat`, for example
`buffered_reply.go`. It should centralize both waiting for authoritative
completion and marker classification so adapters cannot drift.

Suggested shape (names may change during implementation):

```go
type bufferedReply struct {
    Text       string
    Suppressed bool
}

func isNoReply(text string) bool {
    return strings.TrimSpace(text) == "NO_REPLY"
}

func awaitBufferedReply(
    ctx context.Context,
    sub *turn.Subscription,
    idleTimeout time.Duration,
) (bufferedReply, error)
```

`awaitBufferedReply` should:

1. consume subscription events without exposing text to the platform;
2. reset the idle watchdog on every event;
3. finish on `agent_end` or normal event-channel closure;
4. read the authoritative `FinalText` value;
5. return a distinct incomplete-turn error if `FinalText` closes without a
   value;
6. return promptly on context cancellation or idle timeout;
7. classify only the authoritative final value with `isNoReply`.

The helper should not close the subscription implicitly unless ownership is
made explicit in its API. Each adapter currently owns and closes the
subscription, and retaining that rule avoids double-lifecycle ambiguity.

Keep platform operations out of the helper. Typing indicators, Discord
interaction deletion, formatting, chunking, and user-visible error messages
remain adapter responsibilities.

### 3.2 Telegram

Refactor `Bot.streamSubscription` to use the shared buffered helper:

1. start/refesh the typing indicator as today;
2. await authoritative completion without sending deltas;
3. stop typing;
4. on suppression, return without calling `finalizeMessage`;
5. on normal text, retain Telegram formatting and rune-safe chunking;
6. on incomplete/idle failure, retain the current warning behavior.

Telegram already buffers output, so this should be primarily a deduplication
and final classification change.

### 3.3 Discord ordinary messages

Refactor `DiscordBot.streamDiscordSubscription` to buffer like Telegram:

1. keep refreshing Discord typing while pi works;
2. do not create or edit a preview from assistant deltas;
3. await the authoritative final response through the shared helper;
4. stop typing;
5. on suppression, send nothing;
6. otherwise pass final text to the existing Discord Markdown-preserving,
   UTF-16-aware chunk delivery;
7. retain existing incomplete-turn warning behavior.

After migration, remove preview-only state and code where no longer used:
`previewID`, dirty/edit tickers, preview deletion, and preview fallback paths.
Do not remove generic Discord edit/delete API support if interactions or other
features still require it.

### 3.4 Discord application-command interactions

Discord requires an interaction acknowledgement within three seconds, so a
command cannot be completely invisible while work is in progress. Continue to
send the deferred acknowledgement synchronously, but do not stream assistant
deltas into the deferred response.

At completion:

- normal final response: edit the deferred original response and create
  follow-ups for remaining chunks as today;
- `NO_REPLY`: delete the original deferred interaction response so no
  persistent assistant message remains;
- incomplete/idle failure: retain a visible failure response.

Add a `DeleteInteractionResponse` operation to `DiscordAPI`, its real REST
adapter, and the fake API. Use Discord's endpoint:

```text
DELETE /webhooks/{application.id}/{interaction.token}/messages/@original
```

Treat a deletion failure as a logged delivery failure. Do not replace
`NO_REPLY` with visible explanatory text, because that defeats suppression.

A prompt-steer interaction that only acknowledges steering is not the final
assistant response and should retain its existing acknowledgement behavior.

### 3.5 HTTP and turn manager

Do not add marker handling to `turn.Manager`, `httpapi.Server`, SSE encoding, or
the CLI. Their current event stream and authoritative final text must remain
unchanged. This is essential for mixed raw/buffered subscriptions.

### 3.6 Agent system prompt

Update `static/SYSTEM.md` so the agent knows to make its entire response exactly
`NO_REPLY` when no conversational reply should be delivered, especially after a
tool or `wall-e send` has already delivered the intended output. The prompt must
warn against adding Markdown, punctuation, or explanatory text and explain that
Telegram and Discord suppress the marker while raw HTTP callers still observe
it.

## 4. Documentation changes

The initial design documentation is included with this plan:

- `docs/source/channels/index.md` defines raw versus buffered delivery and the
  marker's scope;
- `docs/source/channels/http.md` states that HTTP always exposes raw text,
  including `NO_REPLY`;
- `docs/source/channels/telegram.md` records Telegram's planned silent
  completion behavior;
- `docs/source/channels/discord.md` records removal of previews and the Discord
  interaction caveat.

When implementation lands:

1. remove “planned” / “not implemented yet” wording from those channel pages;

## 5. Tests

### 5.1 Shared helper unit tests

Add table-driven tests for `isNoReply` covering every example in section 2.1,
including surrounding Unicode whitespace. Test `awaitBufferedReply` for:

- multiple text deltas followed by `agent_end` and authoritative final text;
- exact and whitespace-wrapped suppression markers;
- event channel closure with a valid final value;
- event/final closure without a final value;
- context cancellation;
- idle timeout;
- non-text events resetting the idle watchdog;
- authoritative `FinalText` winning over any locally observed/dropped deltas.

### 5.2 Telegram adapter tests

Extend `src/chat/telegram_test.go` to prove:

- exact `NO_REPLY` produces zero `SendMessage` calls;
- whitespace-wrapped `NO_REPLY` produces zero sends;
- prose containing the marker is delivered unchanged;
- lowercase `no_reply` is delivered;
- typing starts and stops for a suppressed turn;
- incomplete and idle turns still send their warning;
- an HTTP/CLI-started Telegram turn suppresses Telegram delivery without
  preventing the raw subscriber from receiving `NO_REPLY`.

### 5.3 Discord ordinary-message tests

Update `src/chat/discord_test.go` to prove:

- no assistant message or edit is made before `agent_end`;
- exact and whitespace-wrapped `NO_REPLY` create no Discord message;
- normal text is delivered only after completion and still chunks at Discord's
  UTF-16 limit;
- prose containing the marker is not suppressed;
- typing runs while buffered and stops at completion;
- incomplete and idle warnings remain visible;
- old preview-edit/delete expectations are removed or rewritten for buffered
  delivery;
- final-delivery failure behavior is tested without relying on a preview
  fallback.

### 5.4 Discord interaction tests

Add tests proving:

- the interaction is deferred before pool/RPC work;
- assistant deltas do not edit the deferred response;
- normal final output edits the original response and uses follow-ups for long
  text;
- exact `NO_REPLY` calls `DeleteInteractionResponse` and performs no final edit
  or follow-up;
- delete failure is logged/contained and does not generate visible replacement
  text;
- prompt-steer acknowledgements remain unchanged.

### 5.5 HTTP regression tests

Extend `src/httpapi/server_test.go` (and CLI tests if useful) with an explicit
regression test whose assistant emits `NO_REPLY`. Assert SSE still contains:

1. `agent_start`;
2. `delta` with `NO_REPLY`;
3. `agent_end`;
4. `done`.

Also cover a mixed-subscriber integration path if the existing harness makes
that practical: raw HTTP receives the marker while the targeted fake buffered
adapter records no external delivery.

## 6. Validation

Run at minimum:

```sh
cd src && go test ./...
make docs
```

If `make docs` is not the repository's current documentation target, use the
Sphinx target documented by the Makefile. Also run `go test -race ./...` when
feasible because subscriptions, typing loops, and interaction completion are
concurrent.

Manual smoke checks after deployment:

1. ask Telegram and ordinary Discord messages to respond exactly `NO_REPLY`;
2. confirm typing stops and no message appears;
3. repeat through a Discord slash command and confirm the deferred response is
   removed;
4. run the same prompt through `wall-e msg http:<id>` and confirm the CLI prints
   `NO_REPLY`;
5. run `wall-e msg telegram:<id>` and `wall-e msg discord:<id>` and confirm the
   CLI sees raw text while the external chat remains silent;
6. verify ordinary non-marker replies arrive only after completion and long
   replies still chunk correctly.

## 7. Acceptance criteria

- Telegram and Discord ordinary messages buffer all assistant text until
  authoritative completion.
- Future client-facing adapters have one documented shared helper/contract to
  reuse.
- Exact trimmed `NO_REPLY` leaves no persistent buffered-channel reply.
- The agent system prompt explains when and how to emit the exact marker.
- Discord interactions retain only the unavoidable temporary deferred state
  and delete it on suppression.
- HTTP/SSE and CLI output remain raw and include `NO_REPLY`.
- Direct sends and transcripts remain unchanged.
- Unit/regression tests cover matching, buffering, mixed subscribers, channel
  failures, and Discord interaction deletion.
- Channel documentation accurately describes the implemented behavior.
